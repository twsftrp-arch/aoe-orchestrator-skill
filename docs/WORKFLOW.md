# 1인 멀티에이전트 워크플로우 — 전체 그림

이 문서는 "구독 여러 개 + 에이전트 세션 여러 개"를 혼자서 굴리는 워크플로우의 설계 이유를 설명합니다. 따라하는 순서는 [SETUP.md](SETUP.md)에 있습니다.

## 스택 한 장

```
┌─ 모바일 관제 ───────────────────────────────┐
│  tailscale + aoe serve (어디서든 함대 확인/지시)│
├─ 오케스트레이션 ────────────────────────────┤
│  HQ 세션 1개 = 오케스트레이터 (이 레포의 SKILL) │
│  감시기(aoe-watch) + 헬스체크(aoe-doctor)      │
├─ 함대 ─────────────────────────────────────┤
│  aoe structured view 워커 N개 (프로젝트당 1개)  │
├─ 구독 포트폴리오 ───────────────────────────┤
│  Claude / ChatGPT(Codex) / Gemini / Cursor …  │
└────────────────────────────────────────────┘
```

## 왜 구독을 여러 개 쓰나

단순히 "많이"가 아니라 **강점 라우팅 + 한도 분산**입니다.

| 역할 | 적합한 구독 | 이유 |
|---|---|---|
| HQ(오케스트레이터) | 큰 컨텍스트 모델 (예: Claude 1M급) | 함대 전체 맥락·긴 운영 기록을 한 세션에 유지 |
| 대량 워커 | 동시 세션이 넉넉한 구독 (예: Codex) | 프로젝트당 1세션을 부담 없이 깔아둠 |
| 보충/사이드 | 나머지 구독 (Gemini/Antigravity, Cursor 등) | 메인 한도 소진 시 대체, 실험·프로토타입 lane |

부수 효과: 한 공급자의 사용량 한도가 막혀도 함대가 멈추지 않습니다. 한도가 풀리는 주기를 알아두고, 막히면 워커를 다른 런타임으로 핸드오프 문서 기반 전환합니다(같은 런타임끼리만 대화 맥락 보존이 되고, 런타임이 바뀌면 핸드오프 문서가 유일한 다리입니다 — 그래서 워커에게 항상 인수인계 문서를 쓰게 합니다).

## 왜 aoe + structured view인가

- 터미널 멀티플렉서(tmux) 화면 긁기로 에이전트 상태를 판독하는 방식은 깨지기 쉽습니다. structured(ACP) 뷰는 이벤트가 DB로 떨어져서 **감시기가 화면이 아니라 데이터를 읽습니다.**
- `aoe acp prompt/history`로 세션 간 메시지 주입·읽기가 명령 한 줄이라, 오케스트레이터가 워커를 프로그래밍적으로 다룰 수 있습니다.

## 왜 오케스트레이터 세션을 따로 두나

워커가 5개를 넘으면 사람이 직접 모든 세션을 폴링하는 비용이 작업 자체보다 커집니다. 오케스트레이터 패턴의 핵심 계약:

- **사람 앞에는 열린 결정 1건만** (보고 큐) — 보고 홍수 방지
- **Decision Readiness** — 증거 갖춘 보고만 받음, "어떻게 할까요?" 금지
- **개입 최소화** — 진행 중인 워커는 건드리지 않음
- **게이트 문서화** — 유휴 워커는 "왜 유휴인지"가 적혀 있어야 함

자세한 규칙은 [SKILL.md](../SKILL.md). 여유가 되면 **standby HQ를 하나 더** 두세요 — 평시엔 인프라 정비와 스킬 상호평가(서로 점수 매기고 보완; 단 "90점+면 종료" 같은 수렴 캡 필수), 메인 HQ 한도 소진 시엔 인수.

## 모바일 관제 (tailscale)

`aoe serve`가 웹 대시보드를 띄우고, tailscale이 그걸 내 기기에서만 접근 가능한 HTTPS로 만들어 줍니다. 외출 중에 폰으로 함대 상태를 보고, 막힌 워커에 결정을 내려줄 수 있습니다. 공인 인터넷에 노출하지 않는 것이 핵심입니다(tailnet 안에서만).

## 운영 리듬 (하루 단위)

1. 아침: `aoe-doctor.sh` 한 번 — 데몬·감시기·함대 생존 확인
2. HQ에게 "순찰 한 바퀴" — 밤사이 완료/막힘 보고 받기
3. 낮: 알림(events.log) 기반으로 HQ가 알아서 분류·보고. 사람은 열린 결정에만 답
4. 사람 손작업(콘솔 클릭, 인증, 실기기 테스트)은 큐의 "손작업" 섹션에 모아뒀다가 시간 날 때 일괄 처리
5. HQ 컨텍스트가 70%를 넘으면 함대 한가할 때 의도적으로 compact

## 비용 감각

- 워커는 유휴 상태에서 토큰을 거의 쓰지 않습니다. 세션을 많이 깔아두는 것 자체는 싸고, 비싼 건 **불필요한 개입과 보고 홍수**입니다 — 그래서 규칙의 대부분이 "언제 침묵할 것인가"입니다.
- 자동화는 "겪은 사고의 재발 방지"까지만. 그 너머(자동 분류기, 상태 머신, 락)는 1인 운영 규모에서 유지보수 대상만 늘립니다. 우리도 평가 루프를 돌려본 결과 그 경계에서 멈췄습니다.
